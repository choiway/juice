# Roadmap: Quickest Path to BEAM Concepts

The goal: get a JavaScript developer from `console.log("hello")` to seeing processes, GenServer, and OTP in action — as fast as possible.

## Phase 1: Minimum Viable Language

The current compiler only handles `console.log("string")`. We need just enough JS to write meaningful demos — not full JavaScript.

**Add in order:**

1. ~~**Variables** — `let`/`const` bindings → Erlang variables~~ **DONE**
2. ~~**`console.log` with expressions** — not just string literals~~ **DONE**
3. ~~**Number literals + basic operators** — `+`, `-`, `*`, `/`, `===`~~ **DONE**
4. ~~**Arrow functions with params** — `(x) => { ... }` → Erlang funs~~ **DONE**
5. ~~**Function calls with arguments** — `greet("world")`~~ **DONE**
6. ~~**String concatenation** — `"hello " + name`~~ **DONE**
7. ~~**`if`/`else`** — → Erlang `case`/`if`~~ **DONE**

**Milestone demo:**
```javascript
const greet = (name) => {
  console.log("hello " + name)
}
greet("beam")
```

This phase is pure plumbing — no BEAM concepts yet, but without it nothing else works.

## Phase 2: Processes (first "wow" moment)

This is where a JS developer sees something they literally cannot do in Node. Expose BEAM processes through familiar-looking built-ins.

**Add:**

1. ~~**`spawn(fn)`** → `erlang:spawn/1` — returns a pid~~ **DONE**
2. **`send(pid, msg)`** → `erlang:send/2` (`!` operator)
3. **`receive(callback)`** → `receive ... end` block (see design note below)
4. **`self()`** → `erlang:self/0`
5. **Atoms as strings** — `"increment"` → Erlang atoms where appropriate
6. **Tuples** — represent as arrays `[pid, "hello"]` → `{Pid, <<"hello">>}`
7. **`for` loop** — `for (let i = 0; i < n; i++)` → Erlang recursion or list comprehension (needed for the 10k processes demo)

### Design note: `receive` semantics

Erlang's `receive` is a blocking selective pattern match — the process *suspends* until a matching message arrives. The callback-style `receive((msg) => ...)` API needs clear rules:

- **Blocking:** `receive` suspends the calling process. It does not return until a message arrives.
- **Single-shot:** Each `receive` call consumes exactly one message, then returns. For continuous listening, the callback must call `receive` again (tail-recursive loop).
- **No pattern matching (yet):** The callback receives the raw message. Selective receive (matching on message shape) is deferred — all messages match.
- **Compilation target:** `receive(callback)` compiles to `receive Msg -> Callback(Msg) end`. For a looping receiver, the compiler must emit a named recursive function.

This is the trickiest mapping in Phase 2. Write the target Erlang output for each demo before implementing.

**Milestone demo:**
```javascript
const pid = spawn(() => {
  receive((msg) => {
    console.log("got: " + msg)
  })
})

send(pid, "hello from another process!")
```

**Killer demo — spawn 10,000 processes:**
```javascript
for (let i = 0; i < 10000; i++) {
  spawn(() => {
    receive((msg) => {
      console.log("process " + i + " got: " + msg)
    })
  })
}
```
Try doing this with threads in Node.

### Design note: crash behavior before supervision

In Phase 2-3, there are no supervisors yet. When a process crashes (e.g., runtime error, bad message):

- The process dies silently by default (standard BEAM behavior).
- The main process should print a stderr message like `Process <0.X.0> exited with reason: <reason>` so developers aren't confused by silent failures.
- This is achievable by having `spawn` link or monitor the child process and log exits — or by setting a custom error logger.
- Phase 4 replaces this with proper supervision, but the Phase 2-3 DX must not leave developers guessing why a process disappeared.

## Phase 3: GenServer

Show JS developers stateful servers with a familiar API. Map object literals to GenServer callbacks.

**Add:**

1. **Object literals** — `{ count: 0 }` → Erlang maps
2. **Property access** — `state.count` → `maps:get`
3. **`GenServer.start(module)`** — → `gen_server:start_link/3`
4. **`GenServer.call(pid, msg)`** — → `gen_server:call/2`
5. **`GenServer.cast(pid, msg)`** — → `gen_server:cast/2`
6. **Convention: object with `init`/`handleCall`/`handleCast`** → GenServer callbacks

### Design note: GenServer callback contract

The convention-based mapping (magic property names → GenServer callbacks) needs a precise contract:

| JS callback | Erlang callback | Arguments | Must return |
|-------------|----------------|-----------|-------------|
| `init()` | `init/1` | none | initial state (any value) |
| `handleCall(msg, state)` | `handle_call/3` | message, current state | `{ reply: <term>, state: <newState> }` |
| `handleCast(msg, state)` | `handle_cast/2` | message, current state | `{ state: <newState> }` |

- If `init` throws, the GenServer fails to start — `GenServer.start()` should return an error or crash.
- If `handleCall`/`handleCast` throws, the process dies (which sets up Phase 4's supervision story).
- Only `handleCall` returns a reply to the caller. `handleCast` is fire-and-forget.

**Milestone demo:**
```javascript
const Counter = {
  init: () => ({ count: 0 }),

  handleCall: (msg, state) => {
    if (msg === "increment") {
      const next = { count: state.count + 1 }
      return { reply: next.count, state: next }
    }
    if (msg === "get") {
      return { reply: state.count, state: state }
    }
  }
}

const pid = GenServer.start(Counter)
console.log(GenServer.call(pid, "increment")) // 1
console.log(GenServer.call(pid, "increment")) // 2
console.log(GenServer.call(pid, "get"))        // 2
```

A JS developer will immediately see: "oh, this is like a stateful service that handles requests one at a time, with no race conditions." That's the lesson.

## Phase 4: OTP Supervision ("let it crash")

The grand finale. Show that the BEAM restarts crashed processes automatically.

**Add:**

1. **`Supervisor.start(spec)`** → `supervisor:start_link/2`
2. **Child specs as objects** → supervisor child specifications
3. **`Process.exit(pid, reason)`** → `erlang:exit/2`
4. **Strategy options** — `"one_for_one"`, `"one_for_all"`
5. **`Supervisor.whichChildren(sup)`** → `supervisor:which_children/1` — returns list of `{id, pid}` pairs

**Milestone demo:**
```javascript
const Counter = {
  init: () => ({ count: 0 }),
  handleCall: (msg, state) => {
    if (msg === "boom") {
      throw new Error("crash!")  // process dies
    }
    if (msg === "increment") {
      const next = { count: state.count + 1 }
      return { reply: next.count, state: next }
    }
  }
}

const sup = Supervisor.start({
  strategy: "one_for_one",
  children: [
    { id: "counter", start: () => GenServer.start(Counter) }
  ]
})

const counter = Supervisor.findChild(sup, "counter")
console.log(GenServer.call(counter, "increment")) // 1
console.log(GenServer.call(counter, "increment")) // 2
GenServer.call(counter, "boom")                    // crashes!

// counter is automatically restarted by supervisor
const counter2 = Supervisor.findChild(sup, "counter")
console.log(GenServer.call(counter2, "increment")) // 1 (fresh state)
```

A JS developer sees a process crash and get restarted with zero intervention. That's the BEAM pitch in one script.

## Execution model

For reference, the compilation pipeline and execution model:

- **JS source** → parsed by `oxc_parser` → **JS AST**
- **JS AST** → compiled by `compiler.rs` → **Erlang source (.erl)**
- **Erlang source** → compiled by `erlc` → **BEAM bytecode (.beam)**
- **Execution:** `erl -noshell -s <module> main -s init stop`

Phase 2+ requires a running BEAM VM (processes, message passing). The `--run` flag already handles this. For Phase 3-4, the generated Erlang module will need to start OTP applications/supervisors before calling `main/0`, which changes the boot sequence — plan for this when implementing GenServer and Supervisor support.

---

## What to skip

These JS features are **not needed** for the demos above and would slow you down:

- Classes, prototypes, `this`
- Promises, async/await
- Module imports/exports (use built-in globals instead)
- Destructuring, spread, rest
- `switch`, ternary, `try`/`catch` (except Phase 4)
- Arrays as data structures (only as tuple syntax)
- Closures over mutable state
- Type coercion edge cases

## Summary

| Phase | New JS Features | New BEAM Concepts | Estimated Complexity |
|-------|----------------|-------------------|---------------------|
| 1 | variables, numbers, arrows, if/else, operators | none (foundation) | Medium — most compiler work lives here |
| 2 | `spawn`, `send`, `receive`, `self`, `for` loop | Processes, message passing | Medium-High — `receive` semantics and `for`→recursion are nontrivial |
| 3 | object literals, property access, GenServer API | GenServer, state management | High — GenServer callback wiring is nontrivial |
| 4 | Supervisor API, `throw` | Supervision, fault tolerance | Low — mostly wiring to OTP, phase 3 did the hard part |

Phase 1 is the most work for the least visible payoff, but it unlocks everything else. Phase 2 is where JS developers will start paying attention. Phase 3 makes them think. Phase 4 closes the sale.
