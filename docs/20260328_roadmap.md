# Roadmap: Quickest Path to BEAM Concepts

The goal: get a JavaScript developer from `console.log("hello")` to seeing processes, GenServer, and OTP in action ‚Äî as fast as possible.

## Phase 1: Minimum Viable Language

The current compiler only handles `console.log("string")`. We need just enough JS to write meaningful demos ‚Äî not full JavaScript.

**Add in order:**

1. ~~**Variables** ‚Äî `let`/`const` bindings ‚Üí Erlang variables~~ **DONE**
2. ~~**`console.log` with expressions** ‚Äî not just string literals~~ **DONE**
3. **Number literals + basic operators** ‚Äî `+`, `-`, `*`, `/`, `===`
4. **Arrow functions with params** ‚Äî `(x) => { ... }` ‚Üí Erlang funs
5. **Function calls with arguments** ‚Äî `greet("world")`
6. **String concatenation** ‚Äî `"hello " + name`
7. **`if`/`else`** ‚Äî ‚Üí Erlang `case`/`if`

**Milestone demo:**
```javascript
const greet = (name) => {
  console.log("hello " + name)
}
greet("beam")
```

This phase is pure plumbing ‚Äî no BEAM concepts yet, but without it nothing else works.

---

## Phase 2: Processes (first "wow" moment)

This is where a JS developer sees something they literally cannot do in Node. Expose BEAM processes through familiar-looking built-ins.

**Add:**

1. **`spawn(fn)`** ‚Üí `erlang:spawn/1` ‚Äî returns a pid
2. **`send(pid, msg)`** ‚Üí `erlang:send/2` (`!` operator)
3. **`receive(callback)`** ‚Üí `receive ... end` block
4. **`self()`** ‚Üí `erlang:self/0`
5. **Atoms as strings** ‚Äî `"increment"` ‚Üí Erlang atoms where appropriate
6. **Tuples** ‚Äî represent as arrays `[pid, "hello"]` ‚Üí `{Pid, <<"hello">>}`

**Milestone demo:**
```javascript
const pid = spawn(() => {
  receive((msg) => {
    console.log("got: " + msg)
  })
})

send(pid, "hello from another process!")
```

**Killer demo ‚Äî spawn 10,000 processes:**
```javascript
for (let i = 0; i < 10000; i++) {
  spawn(() => {
    receive((msg) => {
      console.log("process " + i + " got: " + msg)
    })
  })
}
```
Try doing this with threads in Node.

(This requires adding `for` loops ‚Äî worth it for the demo impact.)

## Phase 3: GenServer

Show JS developers stateful servers with a familiar API. Map object literals to GenServer callbacks.

**Add:**

1. **Object literals** ‚Äî `{ count: 0 }` ‚Üí Erlang maps
2. **Property access** ‚Äî `state.count` ‚Üí `maps:get`
3. **`GenServer.start(module)`** ‚Äî ‚Üí `gen_server:start_link/3`
4. **`GenServer.call(pid, msg)`** ‚Äî ‚Üí `gen_server:call/2`
5. **`GenServer.cast(pid, msg)`** ‚Äî ‚Üí `gen_server:cast/2`
6. **Convention: object with `init`/`handleCall`/`handleCast`** ‚Üí GenServer callbacks

**Milestone demo:**
```javascript
const Counter = {
  init: () => ({ count: 0 }),

  handleCall: (msg, state) => {
    if (msg === "increment") {
      const next = { count: state.count + 1 }
      return { reply: next.count, state: next }
    }
    if (msg === "get") {
      return { reply: state.count, state: state }
    }
  }
}

const pid = GenServer.start(Counter)
console.log(GenServer.call(pid, "increment")) // 1
console.log(GenServer.call(pid, "increment")) // 2
console.log(GenServer.call(pid, "get"))        // 2
```

A JS developer will immediately see: "oh, this is like a stateful service that handles requests one at a time, with no race conditions." That's the lesson.

## Phase 4: OTP Supervision ("let it crash")

The grand finale. Show that the BEAM restarts crashed processes automatically.

**Add:**

1. **`Supervisor.start(spec)`** ‚Üí `supervisor:start_link/2`
2. **Child specs as objects** ‚Üí supervisor child specifications
3. **`Process.exit(pid, reason)`** ‚Üí `erlang:exit/2`
4. **Strategy options** ‚Äî `"one_for_one"`, `"one_for_all"`

**Milestone demo:**
```javascript
const Counter = {
  init: () => ({ count: 0 }),
  handleCall: (msg, state) => {
    if (msg === "boom") {
      throw new Error("crash!")  // process dies
    }
    if (msg === "increment") {
      const next = { count: state.count + 1 }
      return { reply: next.count, state: next }
    }
  }
}

const sup = Supervisor.start({
  strategy: "one_for_one",
  children: [
    { id: "counter", start: () => GenServer.start(Counter) }
  ]
})

const counter = Supervisor.getChild(sup, "counter")
console.log(GenServer.call(counter, "increment")) // 1
console.log(GenServer.call(counter, "increment")) // 2
GenServer.call(counter, "boom")                    // crashes!

// counter is automatically restarted by supervisor
const counter2 = Supervisor.getChild(sup, "counter")
console.log(GenServer.call(counter2, "increment")) // 1 (fresh state)
```

A JS developer sees a process crash and get restarted with zero intervention. That's the BEAM pitch in one script.

## What to skip

These JS features are **not needed** for the demos above and would slow you down:

- Classes, prototypes, `this`
- Promises, async/await
- Module imports/exports (use built-in globals instead)
- Destructuring, spread, rest
- `switch`, ternary, `try`/`catch` (except Phase 4)
- Arrays as data structures (only as tuple syntax)
- Closures over mutable state
- Type coercion edge cases

## Summary

| Phase | New JS Features | New BEAM Concepts | Estimated Complexity |
|-------|----------------|-------------------|---------------------|
| 1 | variables, numbers, arrows, if/else, operators | none (foundation) | Medium ‚Äî most compiler work lives here |
| 2 | `spawn`, `send`, `receive`, `self`, `for` loop | Processes, message passing | Medium ‚Äî new built-ins map directly to Erlang |
| 3 | object literals, property access, GenServer API | GenServer, state management | High ‚Äî GenServer callback wiring is nontrivial |
| 4 | Supervisor API, `throw` | Supervision, fault tolerance | Low ‚Äî mostly wiring to OTP, phase 3 did the hard part |

Phase 1 is the most work for the least visible payoff, but it unlocks everything else. Phase 2 is where JS developers will start paying attention. Phase 3 makes them think. Phase 4 closes the sale.
