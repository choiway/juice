# Roadmap: Quickest Path to BEAM Concepts

The goal: get a JavaScript developer from `console.log("hello")` to seeing processes, GenServer, and OTP in action ‚Äî as fast as possible.

## Phase 1: Minimum Viable Language

The current compiler only handles `console.log("string")`. We need just enough JS to write meaningful demos ‚Äî not full JavaScript.

**Add in order:**

1. ~~**Variables** ‚Äî `let`/`const` bindings ‚Üí Erlang variables~~ **DONE**
2. ~~**`console.log` with expressions** ‚Äî not just string literals~~ **DONE**
3. ~~**Number literals + basic operators** ‚Äî `+`, `-`, `*`, `/`, `===`~~ **DONE**
4. ~~**Arrow functions with params** ‚Äî `(x) => { ... }` ‚Üí Erlang funs~~ **DONE**
5. ~~**Function calls with arguments** ‚Äî `greet("world")`~~ **DONE**
6. ~~**String concatenation** ‚Äî `"hello " + name`~~ **DONE**
7. ~~**`if`/`else`** ‚Äî ‚Üí Erlang `case`/`if`~~ **DONE**

**Milestone demo:**
```javascript
const greet = (name) => {
  console.log("hello " + name)
}
greet("beam")
```

This phase is pure plumbing ‚Äî no BEAM concepts yet, but without it nothing else works.

## Phase 2: Processes (first "wow" moment)

This is where a JS developer sees something they literally cannot do in Node. Expose BEAM processes through familiar-looking built-ins.

**Add:**

1. ~~**`spawn(fn)`** ‚Üí `erlang:spawn/1` ‚Äî returns a pid~~ **DONE**
2. ~~**`send(pid, msg)`** ‚Üí `erlang:send/2` (`!` operator)~~ **DONE**
3. ~~**`receive(callback)`** ‚Üí `receive ... end` block (see design note below)~~ **DONE**
4. ~~**`self()`** ‚Üí `erlang:self/0`~~ **DONE**
5. ~~**Atoms as strings** ‚Äî `"increment"` ‚Üí Erlang atoms where appropriate~~ **DONE**
6. ~~**Tuples** ‚Äî represent as arrays `[pid, "hello"]` ‚Üí `{Pid, hello}`~~ **DONE**
7. ~~**`for` loop** ‚Äî `for (let i = 0; i < n; i++)` ‚Üí `lists:foreach` + `lists:seq` (needed for the 10k processes demo)~~ **DONE**

### Implementation note: `receive` semantics

`receive(callback)` destructures the arrow function at compile time ‚Äî it does **not** compile the callback as a fun. Instead, it extracts the parameter name and body directly:

- `receive((msg) => { console.log(msg) })` compiles to `receive Msg -> io:format("~p~n", [Msg]) end`
- If the callback has no parameters, the pattern variable is `_Msg` (unused)
- **Blocking:** the process suspends until a message arrives (standard Erlang `receive`)
- **Single-shot:** each `receive` call consumes exactly one message then returns
- **No pattern matching:** all messages match ‚Äî selective receive is deferred

**Limitations (to address later):**
- No looping receive yet ‚Äî would need the compiler to emit a named recursive function
- No `receive` with timeout (`after` clause)

**Milestone demo** (working ‚Äî see `examples/messaging.ts`):
```javascript
const pid = spawn(() => {
  receive((msg) => {
    console.log("got: " + msg)
  })
})

send(pid, "hello from another process!")
```

**Killer demo ‚Äî spawn 10,000 processes:**
```javascript
for (let i = 0; i < 10000; i++) {
  spawn(() => {
    receive((msg) => {
      console.log("process " + i + " got: " + msg)
    })
  })
}
```
Try doing this with threads in Node.

### Known limitation: crash behavior before supervision

Currently, spawned processes that crash die silently (standard BEAM behavior ‚Äî no link or monitor). This means:

- If a spawned process hits a runtime error, it disappears with no output
- The main process has no way to detect the failure
- `timer:sleep(100)` at end of `main/0` is a temporary workaround to keep the VM alive long enough for spawned processes to run

**Future improvements:**
- Add `spawn_link` or `spawn_monitor` variants for crash visibility
- Log process exits to stderr so developers aren't confused
- Phase 4's supervision will replace this with proper fault tolerance

## Phase 3: GenServer

Show JS developers stateful servers with a familiar API. Map object literals to GenServer callbacks.

**Add:**

1. ~~**Object literals** ‚Äî `{ count: 0 }` ‚Üí Erlang maps~~ **DONE**
2. ~~**Property access** ‚Äî `state.count` ‚Üí `maps:get`~~ **DONE**
3. ~~**`GenServer.start(module)`** ‚Äî ‚Üí `gen_server:start_link/3`~~ **DONE**
4. ~~**`GenServer.call(pid, msg)`** ‚Äî ‚Üí `gen_server:call/2`~~ **DONE**
5. ~~**`GenServer.cast(pid, msg)`** ‚Äî ‚Üí `gen_server:cast/2`~~ **DONE**
6. ~~**Convention: object with `init`/`handleCall`/`handleCast`** ‚Üí GenServer callbacks~~ **DONE**

### Design note: GenServer callback contract

The convention-based mapping (magic property names ‚Üí GenServer callbacks) needs a precise contract:

| JS callback | Erlang callback | Arguments | Must return |
|-------------|----------------|-----------|-------------|
| `init()` | `init/1` | none | initial state (any value) |
| `handleCall(msg, state)` | `handle_call/3` | message, current state | `{ reply: <term>, state: <newState> }` |
| `handleCast(msg, state)` | `handle_cast/2` | message, current state | `{ state: <newState> }` |

- If `init` throws, the GenServer fails to start ‚Äî `GenServer.start()` should return an error or crash.
- If `handleCall`/`handleCast` throws, the process dies (which sets up Phase 4's supervision story).
- Only `handleCall` returns a reply to the caller. `handleCast` is fire-and-forget.
- Unmatched messages in `handleCall` return `{error, unhandled}` and preserve state (the GenServer stays alive).
- A default `handle_info/2` is always generated to silently drop stray messages sent via `send()`.

### Implementation note: callback compilation

The compiler uses a **wrapper approach** for GenServer callbacks. The JS callback body compiles normally ‚Äî return objects like `{ reply: X, state: Y }` become Erlang maps `#{reply => X, state => Y}`. The generated Erlang wrapper then destructures the result using map pattern matching:

```erlang
handle_call(Msg, _From, State) ->
    __Result = <compiled body>,
    case __Result of
        #{reply := __Reply, state := __NewState} ->
            {reply, __Reply, __NewState};
        _ ->
            {reply, {error, unhandled}, State}
    end.
```

This avoids threading a compilation context through the compiler. The fallback `_` branch keeps the GenServer alive on unmatched messages.

**Limitation:** Callbacks must use `if/else if` chains (not separate `if` blocks with `return`). Separate if-return blocks compile to sequential Erlang case expressions where only the last result is kept. The `if/else if` pattern compiles to nested cases, which is correct.

**Milestone demo:**
```javascript
const Counter = {
  init: () => ({ count: 0 }),

  handleCall: (msg, state) => {
    if (msg === "increment") {
      const next = { count: state.count + 1 }
      return { reply: next.count, state: next }
    } else if (msg === "get") {
      return { reply: state.count, state: state }
    }
  }
}

const pid = GenServer.start(Counter)
console.log(GenServer.call(pid, "increment")) // 1
console.log(GenServer.call(pid, "increment")) // 2
console.log(GenServer.call(pid, "get"))        // 2
```

A JS developer will immediately see: "oh, this is like a stateful service that handles requests one at a time, with no race conditions." That's the lesson.

## Phase 4: OTP Supervision ("let it crash")

The grand finale. Show that the BEAM restarts crashed processes automatically.

**Add:**

1. **`Supervisor.start(spec)`** ‚Üí `supervisor:start_link/2`
2. **Child specs as objects** ‚Üí supervisor child specifications
3. **`Process.exit(pid, reason)`** ‚Üí `erlang:exit/2`
4. **Strategy options** ‚Äî `"one_for_one"`, `"one_for_all"`
5. **`Supervisor.whichChildren(sup)`** ‚Üí `supervisor:which_children/1` ‚Äî returns list of `{id, pid}` pairs

**Milestone demo:**
```javascript
const Counter = {
  init: () => ({ count: 0 }),
  handleCall: (msg, state) => {
    if (msg === "boom") {
      throw new Error("crash!")  // process dies
    }
    if (msg === "increment") {
      const next = { count: state.count + 1 }
      return { reply: next.count, state: next }
    }
  }
}

const sup = Supervisor.start({
  strategy: "one_for_one",
  children: [
    { id: "counter", start: () => GenServer.start(Counter) }
  ]
})

const counter = Supervisor.findChild(sup, "counter")
console.log(GenServer.call(counter, "increment")) // 1
console.log(GenServer.call(counter, "increment")) // 2
GenServer.call(counter, "boom")                    // crashes!

// counter is automatically restarted by supervisor
const counter2 = Supervisor.findChild(sup, "counter")
console.log(GenServer.call(counter2, "increment")) // 1 (fresh state)
```

A JS developer sees a process crash and get restarted with zero intervention. That's the BEAM pitch in one script.

## Execution model

For reference, the compilation pipeline and execution model:

- **JS source** ‚Üí parsed by `oxc_parser` ‚Üí **JS AST**
- **JS AST** ‚Üí compiled by `compiler.rs` ‚Üí **Erlang source (.erl)**
- **Erlang source** ‚Üí compiled by `erlc` ‚Üí **BEAM bytecode (.beam)**
- **Execution:** `erl -noshell -s <module> main -s init stop`

Phase 2+ requires a running BEAM VM (processes, message passing). The `--run` flag already handles this. For Phase 3-4, the generated Erlang module will need to start OTP applications/supervisors before calling `main/0`, which changes the boot sequence ‚Äî plan for this when implementing GenServer and Supervisor support.

## What to skip

These JS features are **not needed** for the demos above and would slow you down:

- Classes, prototypes, `this`
- Promises, async/await
- Module imports/exports (use built-in globals instead)
- Destructuring, spread, rest
- `switch`, ternary, `try`/`catch` (except Phase 4)
- Arrays as data structures (only as tuple syntax)
- Closures over mutable state
- Type coercion edge cases

## Summary

| Phase | New JS Features | New BEAM Concepts | Estimated Complexity |
|-------|----------------|-------------------|---------------------|
| 1 | variables, numbers, arrows, if/else, operators | none (foundation) | Medium ‚Äî most compiler work lives here |
| 2 | `spawn`, `send`, `receive`, `self`, `for` loop | Processes, message passing | Medium-High ‚Äî `receive` semantics and `for`‚Üírecursion are nontrivial |
| 3 | object literals, property access, GenServer API | GenServer, state management | High ‚Äî GenServer callback wiring is nontrivial |
| 4 | Supervisor API, `throw` | Supervision, fault tolerance | Low ‚Äî mostly wiring to OTP, phase 3 did the hard part |

Phase 1 is the most work for the least visible payoff, but it unlocks everything else. Phase 2 is where JS developers will start paying attention. Phase 3 makes them think. Phase 4 closes the sale.
