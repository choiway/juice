# Why Rust for Juice

Juice compiles JavaScript to the Erlang BEAM. We evaluated Rust, C, Zig, Erlang/Elixir, and OCaml as implementation languages. We chose Rust.

## The deciding factor: JavaScript parsing

Full JS/TS parsing is a massive spec surface — ESM, JSX, async/await, destructuring, optional chaining, generators, and more. Writing a compliant parser from scratch is easily 6+ months of work.

Rust has **oxc_parser** — a production-grade, spec-compliant JS parser used by Biome and Rolldown. It gives us a complete AST for free. No other language has a comparable off-the-shelf JS parser library.

## Other advantages

- **Type system**: Algebraic data types and exhaustive pattern matching are well-suited for compiler internals (ASTs, IRs, node type dispatch).
- **Performance**: The resulting compiler is fast. While not critical for v0, it matters as the language grows.
- **Ecosystem**: Large community, easy to find contributors, strong tooling (cargo, clippy, rust-analyzer).

## What we considered

| Language | JS Parser Available? | Notes |
|----------|---------------------|-------|
| **Rust** | Yes (oxc, swc) | Best ecosystem for this problem |
| **Zig** | No | Would require writing a parser from scratch; pre-1.0 |
| **C** | Partial (quickjs internals) | Manual memory management painful for tree/graph structures; no sum types |
| **Erlang/Elixir** | No | Native BEAM toolchain access is nice, but no JS parser |
| **OCaml** | No (Flow is a reference but not reusable) | Great for compilers, but smaller ecosystem |

## Architecture note

We target **Erlang Abstract Format** (emitting Erlang source text in v0), not raw BEAM bytecode. This is the same approach used by Elixir, Gleam, and LFE. The Erlang compiler handles optimization and bytecode emission via `erlc`.
