# Box: Interactive REPL

## Overview

`juice box` launches an interactive REPL (read-eval-print loop) for Juice, similar to Elixir's `iex`. It lets you type JavaScript expressions and see them executed on the BEAM immediately.

```
$ juice box
box v0.1.0 - juice interactive shell
Type a JS expression. Ctrl+D to exit.

box> console.log("hello, world")
hello, world

box> ^D
Goodbye!
```

## How it works

The REPL reuses the same parse-compile pipeline as file compilation, but skips writing `.erl` files and `.beam` assembly. Instead, compiled Erlang expressions are executed directly via `erl -eval`.

```
                  REPL loop (per input line)
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  user     │    │  JS AST  │    │  Erlang  │    │  erl     │
│  input    │───>│  (oxc)   │───>│  exprs   │───>│  -eval   │
│          │    │          │    │          │    │  -noshell │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
   stdin        oxc_parser      compile_expr    direct exec
```

Compared to file compilation which goes through `erlc` and produces `.beam` files, the REPL evaluates expressions in-process using the Erlang VM's `-eval` flag. The trade-off is ~100-200ms Erlang VM startup per input line, which is acceptable for interactive use.

## Implementation

### Files

- **`src/repl.rs`** — The REPL loop. Reads lines from stdin, parses with oxc, compiles each expression statement via `compiler::compile_expr()`, then executes with `erl -eval '<exprs>, halt().' -noshell`.
- **`src/compiler.rs`** — Added `pub fn compile_expr()` to expose expression-level compilation. The existing `compile()` function wraps expressions in a full Erlang module; `compile_expr()` returns raw Erlang expression strings (e.g., `io:format("hello~n")`).
- **`src/main.rs`** — Added subcommand dispatch: `juice box` calls `repl::run()`, anything else falls through to file compilation.

### Key design decisions

**`erl -eval` instead of temp files.** The file compilation path writes `.erl` → runs `erlc` → produces `.beam`. For the REPL, we skip all that and pass Erlang expressions directly to `erl -eval`. This avoids temp file management and is simpler for single-expression evaluation. Rust's `Command` API passes arguments via `execvp` (not through a shell), so double quotes in `io:format("...")` are safe without escaping.

**New allocator per iteration.** oxc uses an arena allocator that owns all AST nodes. The REPL creates a fresh `Allocator` each loop iteration so memory from previous inputs is freed. Since `compile_expr` returns owned `String` values (not references into the AST), the allocator can be dropped safely after compilation.

**Multiple expressions per line.** If the user types `console.log("a"); console.log("b")`, oxc parses two statements. The REPL compiles each one and joins them with commas for the `-eval` argument: `io:format("a~n"), io:format("b~n"), halt().`

### Current scope

The REPL supports the same expressions as the compiler — currently just `console.log("string")`. As the compiler gains support for more JS constructs, the REPL will pick them up automatically through `compile_expr()`.
